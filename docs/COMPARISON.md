# Comparison with LogClusterC and LogCluster Perl

DeltaZulu.LogCluster is inspired by the LogCluster algorithm and existing implementations, but it is an independent C# implementation for parser suggestion and DeltaZulu.Platform integration. This document records the intentional compatibility points and divergences from the C and Perl versions so maintainers can compare outputs without assuming this project is a direct port.

## Relationship to LogClusterC

DeltaZulu.LogCluster is inspired by Risto Vaarandi and Mauno Pihelgas's LogCluster algorithm and by [LogClusterC](https://github.com/zhugegy/LogClusterC), the C reference implementation, but it is an independent reimplementation, not a port. The core identity model is deliberately the same; several supporting heuristics are deliberately narrower. This section records where and why, so the gap is a documented decision rather than a latent surprise.

### What matches

* **Candidate identity.** Both tools identify a pattern by the ordered sequence of "frequent" anchor words in a record, with runs of infrequent words between anchors folded into a `[min, max]` word-count envelope rather than enumerated. LogClusterC stores this as a newline-joined string key plus a `fullWildcard[]` array on `struct Cluster`; DeltaZulu.LogCluster stores it as a `CandidateKey` over token IDs plus `GapStatistics` per gap on `PatternCandidate`. The identity semantics are the same even though the storage differs.
* **Support counting.** In LogClusterC, `Elem.count` for a word counts the number of distinct lines containing that word; a repeat within one line is explicitly not double-counted, and a cluster's `count` counts distinct matching lines the same way. `LogClusterMiner.DiscoverFrequentWords` and `PatternCandidate.ObserveSupport` use the same "one record, one increment" rule via a per-pass stamp and last-sequence-number guard, so support numbers are comparable in meaning even though the underlying data structures differ.
* **The same threshold gates both word frequency and candidate survival.** LogClusterC's `--support`/`--rsupport` value is used both in `step_1_find_frequent_words` and in `step_3_find_clusters_from_candidates`. `LogClusterOptions.MinSupport` plays the identical dual role here.
* **A wildcard is only rendered when the observed gap can be non-empty.** LogClusterC's `print_cluster` omits `*{min,max}` entirely when the maximum observed width at that position is `0`. `PatternCandidate.RenderLogCluster` and `PatternCandidate.RenderRule` do the same with a `gap.MaxWords > 0` guard, rather than printing a degenerate `*{0,0}`.

### Where we deliberately diverge

* **No prefix-trie aggregate-support rollup.** LogClusterC's `--aggrsup` heuristic builds a prefix trie of every candidate and, for each wildcard candidate, walks the trie to find all more-specific candidates consistent with its wildcard envelope across arbitrarily many gap positions simultaneously, summing their support into the general candidate. `LogClusterMiner.MergeShiftedCandidates` only handles the single-token, single-edge case: a candidate whose anchor sequence is exactly one token longer than another's, with the extra token at the leading or trailing edge. It is a narrow analog, not a port; chained, multi-position, or interior differences are intentionally left as separate candidates rather than rolled up. Treat any `--aggrsup`-style total-support-rollup expectation as out of scope until this is revisited as a real feature.
* **No word-dependency-matrix join-clusters heuristic.** LogClusterC's `--wweight` computes, for every pair of frequent words, a corpus-wide conditional-probability matrix, `dep(w1, w2) = matrix[w1][w2] / matrix[w1][w1]`, then marks an anchor position as a token, or mergeable position, when its average dependency on the cluster's other anchors falls below a threshold. `LogClusterMiner.MergeLowDiversityVariants` uses a much cheaper, purely local proxy instead: candidates that agree everywhere except one anchor position are merged when the number of distinct values at that position is small relative to their combined support (`WordWeightThreshold`). This catches the common case the option is meant for, such as a handful of recurring source IP values, without the `O(words²)` matrix, but it does not consider inter-word dependency the way `--wweight` does, and the two will disagree on corpora where that distinction matters.
* **Tokenization splits on all Unicode whitespace, not just spaces.** LogClusterC's default word delimiter is the regex `[ ]+`, meaning ASCII space only; tabs are literal characters unless `--separator` is overridden. `TokenizedRecord.Tokenize` always splits on `char.IsWhiteSpace`, so tab- or mixed-whitespace-delimited fields, such as `user1\tlogin\tsuccess`, are split into separate anchors or gap words by default, with the original separator preserved for rendering. There is no equivalent of `--separator` to opt back into space-only splitting.
* **No per-line word or length caps.** LogClusterC caps a line at `MAXWORDS` (512) words and `MAXLINELEN` (10,240) bytes; a line with more words than that is silently truncated. DeltaZulu.LogCluster has no equivalent cap on words or line length per record. The only limits are the corpus-wide `--max-records` and `--max-input-bytes` options.
* **No sketch-based pre-filtering.** LogClusterC's optional `--wsize`/`--csize` sketches are single-hash counting arrays used as a cheap pre-pass to avoid materializing hash-table entries for words or candidates that provably cannot meet the support threshold, at the cost of an extra file pass. DeltaZulu.LogCluster has no sketch pre-pass; instead it chooses between materializing all records or re-reading the source once per pass to manage memory.
* **No word-filter/search/replace preprocessing.** LogClusterC's `--wfilter`/`--wsearch`/`--wreplace` normalize individual words before they participate in frequency counting, such as collapsing `key=value123` to `key=VALUE`, with the original word preferred over the normalized form when both are frequent. DeltaZulu.LogCluster has no equivalent preprocessing step. One consequence worth noting: LogClusterC's own outlier pass (`outliers.c`) does not apply this normalization even though candidate generation does, so a word normalized into a frequent form during mining can fail to match during outlier detection. Since DeltaZulu.LogCluster has no normalization step, it has no equivalent inconsistency to reproduce or fix.
* **Deterministic output ordering.** LogClusterC's default output order is an unstable `O(n²)` selection sort by support, and its arity-ordered mode (`--outputmode 1`) leaves ties in hash-bucket order; neither is a fully deterministic tie-break. `LogClusterMiner.Mine` orders candidates by a fixed, fully deterministic key: score descending, then support descending, then specificity descending, then pattern text ordinal ascending.
* **An original multi-factor score.** LogClusterC ranks candidates by raw support or, in arity mode, by anchor count with no combined score. `CandidateScorer` computes a weighted blend of support strength, anchor quality, gap consistency, and pattern specificity, with weights exposed as CLI options. This has no LogClusterC counterpart; it is original to this project, not a port of anything in the reference tool.
* **Liblognorm rule rendering and gap-motif detection are new.** `LiblognormMotifs` and `PatternCandidate.RenderRule` have no LogClusterC counterpart at all. The C tool only emits LogCluster-style patterns, not liblognorm parser rules.

These divergences were deliberate simplifications made when the miner was designed, not oversights. Language elsewhere in this codebase that says `MergeShiftedCandidates` or `MergeLowDiversityVariants` mirrors LogClusterC behavior should be read as "narrow analog of," not "faithful port of."

## Deviations from the original LogCluster Perl algorithm

This project is a reinterpretation of Risto Vaarandi's `logcluster.pl`, not a line-for-line port. The core two-pass idea, finding frequent words and then grouping records by their frequent-word tuple, carries over, but several pieces are deliberately narrower, deliberately new, or simply not implemented. Anyone comparing output against the Perl tool on the same input should expect different candidate sets and different rankings, not just different formatting.

### Candidate merging is a narrower approximation of `--aggrsup` and `--wweight`

`logcluster.pl`'s `--aggrsup` uses a prefix tree, implemented by `build_prefix_tree` and `find_more_specific`, to fold a candidate's support into any other candidate whose word sequence is a superset compatible with the general candidate's observed gap-size bounds. Extra anchor words can appear inside any gap, not only at the edges, and more than one extra word is fine as long as the sizes fit. `MergeShiftedCandidates` here only merges a candidate into a base that is exactly one anchor shorter with the extra anchor at a leading or trailing edge; internal insertions and multi-token differences are never merged this way.

Likewise, `--wweight` computes real word-dependency statistics, `dep(a,b)`, using four weight functions, `f1` through `f4`, over the whole word co-occurrence matrix and joins clusters whose low-weight words differ. `MergeLowDiversityVariants` approximates only the single-position case: candidates differing at exactly one anchor position are joined when the number of distinct values at that position is small relative to their combined support (`WordWeightThreshold`). There is no dependency matrix and no multi-word generalization. Both merges are also mutually exclusive per candidate to avoid composing a stale edge merge with a later position merge.

### Ranking is a novel weighted score, not a support ordering

`logcluster.pl` prints clusters ordered purely by `Count`, or support, descending. This project introduces `CandidateScorer`, a weighted combination of support strength, anchor quality, gap consistency, and pattern specificity that has no equivalent in the original algorithm or its papers. The `--weight-*` flags exist specifically because this scoring model is an invented ranking heuristic, not a reproduction of published behavior.

### Liblognorm rule rendering and parser-motif suggestion are new

`LiblognormMotifs` and the rule-rendering half of `PatternCandidate.ToOutput` have no counterpart in `logcluster.pl`, which only emits `*{min,max}` patterns and a support count. Suggesting `ipv4`, `number`, `word`, and other parsers, then distinguishing executable rules from structural sketches, is functionality this project adds on top of the mining algorithm.

### Missing preprocessing and vocabulary features

The Perl tool supports line filtering or rewriting before clustering with options such as `--lfilter`, `--template`, and `--lcfunc`, and infrequent-word generalization into word classes with options such as `--wfilter`, `--wsearch`, `--wreplace`, and `--wcfunc`. Neither exists here; input records are mined as-is. A custom word separator (`--separator`) is also not supported. Tokenization always splits on Unicode whitespace runs, matching the Perl default rather than every Perl configuration option.

### Missing large-input approximation and iteration features

The Perl tool can build hash-bucket sketches (`--wsize`/`--csize`) to bound memory when exact per-word or per-candidate counters would be too large, and can dump or reload clusters and word-dependency data (`--writedump`/`--readdump`, `--readwords`/`--writewords`) to re-evaluate weight thresholds without re-mining. DeltaZulu.LogCluster instead bounds memory with a materialize-versus-stream strategy choice and hard `--max-records`/`--max-input-bytes` limits; there is no approximate counting and no saved intermediate state to iterate on.

### Gap statistics are collected in a dedicated third pass, only for survivors

`logcluster.pl` updates each candidate's wildcard min/max size in the same pass that discovers candidates, for every candidate regardless of whether it will ultimately survive the support threshold. Here, `PatternCandidate.InitializeGaps` and `PatternCandidate.ObserveGaps` run in a separate `CollectEvidence` pass over only the candidates that survive merging and `MinSupport` filtering. This avoids tracking gap samples, separators, and parser votes for candidates that get discarded, at the cost of one additional full pass over the input.

### Output ordering is fully deterministic; the Perl tool's is not

Perl's hash key iteration order, such as `keys %candidates` and `keys %clusters`, is randomized per process by design, so ties in support-only ordering can vary between runs on identical input. Final output here is sorted by a total order: score, then support, then specificity, then pattern text. Re-running on the same input always produces the same candidate order.
